如何快速融入一个新的团队

发表于 2017-06-25 | 分类于 随感

在互联网行业程序员跳槽是很常见的，虽然不鼓励频繁换工作，但要在一家公司连续工作五年以上的确很难，尤其是对刚毕业的大学生来讲。那么如果你换了一份新工作，作为一名技术开发人员如何快速进入角色，融入团队，在短时间内让自己进入高效的工作状态呢？结合我自己身边的一些例子（有正面也有反面）来简单谈一谈自己的看法。

首先，先把思想转变过来

1，“这项目代码怎么这么烂！”

  • 作为一个新人刚看到项目代码的第一天如果大声说出这样的话，我觉得是非常伤害同事之间感情的，因为你不知道项目经过了多少的版本迭代，又或者出于什么考虑项目代码要这么设计，你只是单纯地觉得不符合你的编码习惯，就扣了一个“烂代码”的帽子，也直接否定了老同事之前工作和努力，我觉得这并不负责任。你可能觉得怎么可能会有人当众说这种话呢，但是，现实中就是有这种例子，我亲身经历的。

    从另外一个角度来说，公司既然招了新员工加入团队，就是要让你来解决问题的，从入职的那一刻起，你就是团队中的一员了，这就是你自己的项目了。好比之前有一个故事，一个新娘子嫁给一个书生，晚上入寝前发现家里竟然有老鼠，新娘说：”你们家怎么还有老鼠呢，你之前可没跟我说过啊“。第二天早上起来，新娘子又看到了老鼠，一把抓过鞋子朝老鼠扔过去：“可恶，让你偷吃我们家大米”。

2，”天啊，你们怎么不用 XX 框架，这么流行的框架你们竟然不用…“

  • 这种事吧，我觉得团队当时既然选择了某某技术方案，就有当时的历史原因，我们刚进入到一个技术团队的时候，不要上来就是来一个否定连环炮，让别人觉得很尴尬。我觉得进入团队的第一件事，是从心态上归零。像在开发岗位上工作了好多年的程序员，一般在代码风格，结构，框架上都有了自己的习惯，这些习惯不说好坏，如果和新团队的风格和习惯存在冲突，必然会影响到团队对自己的第一印象，降低融入感，甚至影响整个团队的开发效率，让别人觉得有一种被入侵的感觉。所以我建议，加入一个新团队伊始，先忘记自己那一套东西，先去学习和了解，先融入再做调整和建议，切忌脚踏七彩祥云，自带 BGM 登场。

其次，融洽愉悦的同事关系是协作的基础

  • 我一直都觉得如果同事之间能把关系相处好，那么工作开展起来就会很顺利。那么和同事相处好关系呢，我觉得一个很重要的因素就是信任，如果一个团队里面一个资深的前端和一个经验不是那么丰富的后端一起配合，很大概率前端就要吐槽后端的接口设计了。如果发生这种情况带着情绪工作也对自己没有什么帮助，倒不如把自己的一些经验告诉对方，相信只要你说的有道理，他一定会感激并信任你的。

  • 私下沟通往往会起到很好的效果。打个比方，如果你觉得后端开发人员设计的接口不是那么的好用，那你不如中午和他一起吃个午饭，吃饭的时候顺便沟通一下这个事情，大家都是为了公司做事，相信他也不会说那么不近人情。

最后才是技术代码层面

  • 阅读现有代码，理解代码框架，暂时忘记自己以前那一套。作为有经验的工程师，一般团队都预期你能快速上手参与到项目中，最多一个星期就要参与到实际的编码工作了。此时重中之重是让自己的代码悄无声息的融入到团队的代码中，让团队的其他人在 review 你代码时看不出是你所写，感觉不到你是新加入者。在代码框架上不同团队的风格迥异，有些 MVP，有些 MVVM，有些注重模块交互，有些采用分层式架构，如果写出破坏现有结构的代码，在团队老手眼里会显得十分突兀，被面对面培训教育就更尴尬了。对 review 其他成员的代码，在风格上也尽量保持统一，最大限度的隐藏自己的代码。程序员之间的信任与好感，没有比代码认同来的更快了。

  • 了解工程的基础设施，不要重复造轮子浪费时间。拿 iOS 开发来说，任何一个项目中肯定有很多封装好的工具类，比如字符串处理类的，图片圆角处理类，加解密工具，或者有自己的一套私有库，俗称轮子。每个工程师手头都有一些自己用得顺手，甚至是自己造的轮子。新加入的团队必定也有一套基础设施库了，大公司一般都有专门的造轮子团队，比如类似 SDWebImage 这中图片请求库，还有内存管理库，数据持久化库，网络库，甚至是细到 NSString 的处理（比如 base64，urlencode 等）都有规定的轮子。在动手开发之前不妨看一下项目或者询问一下旁边的小哥，这个功能是不是已经有了现成的工具类，免得重复造轮子。

  • 一次性理清楚项目开发流程。每个团队的项目都有自己的迭代演进方式，从产品经理处获得需求，到 jira 上查看 task 列表，完成编码提交测试，通过禅道查询 bug 反馈，修复测试，如此循环，持续集成，产品上线。整个流程一般都会有个内部系统来完成，在进入新团队的前期，可以把自己的问题集中一下，然后找个适当的时机，找个对项目流程比较清楚的人全套了解一遍。
